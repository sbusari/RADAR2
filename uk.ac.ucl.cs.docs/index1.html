RADAR: A Lightweight Tool for Requirements and Architecture Decision Analysis

Saheed A. Busari and Emmanuel Letier

Overview

Uncertainty and conflicting objectives make many requirements and architecture decisions particularly hard. Quantitative probabilistic models allow software architects to analysesuch decisions using stochastic simulation and multi-objective optimisation, but the difficulty of elaborating the models is an obstacle to the wider adoption of such techniques. To reduce this obstacle, we present a novel modelling language and analysis tool, called RADAR, intended to facilitate requirements and architecture decision analysis. The language has relations to quantitative AND/OR goal models used in requirements engineering and to feature models used in software product lines. It, however, simplifies such models to a minimum set of language constructs essential for decision analysis. The paper presents RADAR's modelling language, automated support for decision analysis, and our experience of applying the language and tool to four real-world examples.

RADAR is a java based tool designed to be similar to simple equations that software architects use for backof-the-envelope calculations. RADAR, however, provides sophisticated analysis that cannot be performed through backof-the-envelope calculations: it allows analysing uncertainty through Monte-Carlo simulations, shortlisting Pareto-optimal solutions through multi-objective optimisation, and computing expected value of information that can be used to decide whether to seek more information or perform a model detailed analysis before making a decision

The main contributions of this paper are:

  • The RADAR modelling language, that enables decision makers to write domain specific requirements and architecture decision models (which contains elicited stakeholders' goals and quality variables which includes AND-Refinement, OR-Refinements and model parameter estimates elicited either from domain experts or statistical data etc).
  • The automated analysis technique for analysing uncertainty through Monte-Carlo simulations, shortlisting Pareto-optimal solutions through multi-objective optimisation, and computing expected value of information that can be used to decide whether to seek more information or perform a model detailed analysis before making a decision
  • The open-source RADAR tool, which we make freely available  
  • An extensive evaluation of the expressiveness of the modelling language and applicability of the analysis approach and prototype tool on four real-world case studies

Modelling Language

RADAR allows decision makers to explore alternative designs using quantitative probabilistic models which helps to clarify decision problems and adequately captures the stakeholders' goals and correctly predicts the impacts of decisions on these goals. A RADAR model consists of a set of objectives and quality variables definitions:
  1. Objective Definition, which has the form:
    Objective Name = (Min| Max) Statistic (X)
    where Statistic (X) is a statistical measure on a single quality variable X. Statistical measure include:
    EV(X) meaning expected value of variable X.
    Pr(X ~ x) meaning the probability that X ~ x, where ~ could be any of <, >, ≤, ≥, =.
    percentile(X, i) meaning the the i th percentile of X i.e the value of x such taht Pr (X ≤ x) = i.
  2. Quality Variable definitiions , which could be either an AND-refinement, OR-Refinement or parameter estimation:
    An AND-Refinement is of the form:
      X = F (X1...Xn);
    where X is a quality variable and F (X1...Xn) is an arithmetic experession or Boolean expression involving quality variables X1...Xn.
    An OR-Refinement has the form:
     X = decision (label){
      Option1 : Expression1;
      ....
      Optionn : Expressionn;
      }
    where label is the decision name, Optioni are option names and Expression i is an AND-refinement or parameter estimation defining the value of X if Optioni is selected.
    A parameter estimation has the form:
     X = ProbabilityDistribution
    where ProbabilityDistribution defrines a probability distribution for the quality variable X. Probability distribution include:
    uniform(min, max) representing a uniform distribution between min and max.
    triangular(min, mode, max) representing a triangular distribution with lower limit min, upper limit max and mode.
    normal(lowwer, upper) representing a normal distribution characterised by the lower and upperbounds.
    normalCI(lowwer, upper) representing a normal distribution characterised by the lower and upperbounds of its 90% confidence interval (i.e lower is the 5th percentile and upper the 95th percentile).
    deterministic(x) representing that the variable has teh certain value x.

Model Analysis

Figure 1 depicts the high level architecture of RADAR analysis component. It has three main parts, namely, the simulator, optimiser and information value analyser. Before RADAR performs model analysis, software designers will have to write a decision model, in a file with an extension rdr, and according to RADAR's defined syntaxes. RADAR then takes this file as input and then parses it to generate a semantic model, from which model elements (such as objectives, quality variable and parameters) can be interpreted for analysis. The analysis results are stored in a user-specified output directory. This directory consist of a .out and .csv files that store the analysis result, such as, the optimisation analyis and infromation value analysis. It also contains the figure folder, where pareto plot is stored, and a graph folder that stores the AND/OR refinement graph between quality variables and decision dependency graphs in DOT format, to ve viewed using Graphviz. Below is a brief description of the three main analysis components of RADAR.

  1. The Simulator provides two related functions: (i) given a candidate solution s, it returns the values of all objectives for s; (ii) given a set S of candidate vector X that contains simulations of X and a simulation matrix matrixVarP that contains the simulations for all model parameters used to compute X.
  2. The Optimiser returns a shortlist of Pareto-optimal solutions together with their objective values for a given decision model.
  3. The Information Value Analyser returns the expected total and partial perfect information ( EVTPI and the EVPPI ) of all non-deterministic model parameters with respect to the given objective and set of solutions.

The tool is implemented in Java. It uses ANTLR to generate the model parser and generates diagrams in the DOT format that can be visualised and converted to other formats using Graphviz

Subject Models

We present four real world architecture decision models for four different systems from the literature:

  1. The Situation Awareness System (SAS) Model was developed for a system that supports the deployment of personnel in emergency response scenarios such as natural disasters or large scale riots. SAS applications would run on Android devices carried by emergency crews and would allow them to share and obtain an assessment of the situation in real-time (e.g., interactive overlay on maps), and to coordinate with one another (e.g., send reports, chat, and share video streams. We have developed the SAS model based on the previously identified set of design decisions, options and goals to be achieved by this system by a team of academics and engineers from a government agency.

  2. The NASA ECS model objectives, decisions and data obtained from 2002 SEI Report "Making Architecture Design Decisions: An Economic Approach". Note: CBAM estimate individual impact of architectural strategies on utility but not their combined impacts. Modelling combined impact requires richer application specific model.
  3. The Plastic Card Fraud Detection Model was developed for a system that detects when plastic card accounts have been compromised by fraudsters who are using the account to steal funds. Design decisions for fraud detection systems include:
    • the processing type that can be continuous or batch; continuous processing analyses transactions individually, whereas batch processing analyses sets of transactions together;
    • the fraud detection method which can be a *two-class supervised classification method* in which a classifier is trained from past samples of fraudulent and non-fraudulent transactions, or a *non-statistical rule-based method* that flags transactions matching configurations known to be high risk;
    • When a classifier fraud detection method is chosen, the alert threshold above which the classifier should flag a transaction as suspescted.
    • the blocking policy that can include blocking an account as soon as the fraud detection algorithm flags a transaction as fraudulent, or only blocking the account after the suspected fraud has been confirmed by human investigators.
    These design decisions have impacts on how quickly frauds are detected and compromised accounts blocked and on the cost-effectiveness of the fraud detection system. The optimisation of plastic card fraud detection systems includes two main objectives: maximise the expected benefit and minimise investigation load
  4. The Building Security Policy Decisions Model analyses security policy decisions for an organisation concerned with leaks of confidential information. The organisation is concerned about its employees' habit of sharing documents through a globally shared drive. Having files on the shared drive increases the risk and potential damages caused by insiders' leaks. The organisation is considering recommending its employees to share documents through emails or portable media devices instead of using the shared drive but this recommendation may create other security risk and may not be followed by employees. The organisation is also concerned by loss and theft of documents stored on mobile phones, laptops and portable media devices. Theft of mobiles devices and portable media can happen outside and inside the organisation's building. To prevent thieves entering the building, the building entrance is equipped with automated gates that require employee to swipe their access card to access the building. Some employees, however, let other persons tailgate them when entering the building, for example to let other employees in when they have forgotten their access card. The organisation is considering adding a security guard at the building entrance to prevent tailgating. It is hoped this will reduce the risk of thefts inside the building although the actual impact is uncertain. Preventing all tailgating may annoy employees as they will waste time to enter the building when they have forgotten their access card. The primary decision objectives are related to the uncertain costs associated to the disclosure of confidential documents
